# 暂缓的交易策略与功能建议

这些是已经讨论过、但当前决定先跑一段时间再看的方向。它们不是当前待办，也不代表必须实现；主要用于后续复盘数据足够后重新评估。

## 当前观察期优先事项

先观察新上线链路是否有效：

- `pending_management` 是否能给出具体、可执行的取消/修改建议，而不是泛泛提醒。
- `entry_review_tasks` 的 6h / 24h / 48h 复查是否过密或过少。
- `quality_score` 与主观判断、后续真实收益是否一致。
- AI 是否因为 active pending、复盘结论和质量分变得更克制，减少重复 pending 与追价。

## 策略层：后续可评估

### 1. 月内阶段切换策略

把月初、月中、月末的操作风格做得更硬：

- 月初：允许更积极寻找高质量建仓机会。
- 月中：等待确认，减少第一次快速逼近 barrier 时逆势抄底。
- 月末：优先收割、保护利润和撤掉低质量 pending，不再用同一套标准追新仓。

适合等新复盘数据积累后，结合不同阶段的真实盈亏再调。

### 2. 利润锁定规则

当本月已实现收益或净值进度达到目标的 50% / 70% / 90% 时，自动切换风险档：

- 降低新增中价 No、低价 Yes 的上限。
- 提高止盈优先级。
- 撤掉或缩小激进 pending。
- 接近目标后优先防守，避免把利润还回去。

这个方向最直接服务“稳定每月 20%”，但需要先观察现有月度目标和复盘层是否足够有效。

### 3. 更硬的相关性上限

同方向、同距离带、同阶段的多个 token 本质上是同一个 BTC 路径赌注。后续可考虑：

- 同一 `correlation_group` 设置总风险上限。
- 同方向 above / below 分别设置总仓位上限。
- 相邻 strike 之间做组合级限额，而不是只看单 token。

### 4. 亏损后冷却期

某类入场亏损后，短时间内不要只靠提醒，而是设置冷却：

- 同类组合当月连续亏损 2 次后，24-72 小时内只观察。
- 需要明确确认信号才允许解除冷却。
- 可按 tier、distance bucket、entry gate 或组合 key 触发。

### 5. 交易频率控制

月度市场不适合过度高频调整。后续可限制：

- 每日新增风险敞口次数。
- 每日新增 USDC 上限。
- 波动大时连续补仓次数。
- 同一方向连续新增 pending 的次数。

## 执行与复盘层：后续可评估

### 6. 把止盈纪律升级为计划

当前止盈更多是建议。后续可以让新买入自动生成更结构化的止盈计划：

- 到达 20%-30% 收益先卖一部分。
- 临近月末降低目标收益，优先落袋。
- 接近 barrier 或 BTC 动量反向时自动提示减仓。

先不做自动化，避免在质量分和复查机制还未校准前过度交易。

### 7. 月内亏损归因熔断

不只看总亏损预算，而是看亏损来自哪类组合：

- 中价 No 连续亏损。
- 低价 Yes 无催化亏损。
- 下方 dip No 第一次逼近 barrier 时逆势买入亏损。
- 追价后买入亏损。

某类组合超过阈值后，当月暂停同类新单，直到下月或人工解除。

### 8. 价格执行质量追踪

状态：已进入实现 / 观察期。

记录建议价、触发时市场价、实际成交价、成交后 1h / 6h 表现。用于区分：

- 策略判断错。
- 触发条件太差。
- 成交价太差。
- 追价导致 edge 被吃掉。

实现入口：

- `execution_quality_trades` 记录 manual pending 触发下单时的建议价、limit、下单前盘口、实际成交价和滑点。
- `execution_quality_snapshots` 记录成交后 1h / 6h 的价格表现。
- Dashboard 的 pending 面板展示最近记录，先作为观测数据，不直接驱动自动交易。

这类数据积累后，可以反向优化 pending 触发价和 price spec。

## 新功能深化：暂缓

这些是当前已上线新功能的自然深化，但先不急着做：

- Pending 修改前后 diff 确认：展示 old → new，而不是只展示 JSON。
- 把 due / overdue `entry_review_tasks` 带回下一轮 AI prompt，让 AI 主动回答复查结论。
- 用 1-2 周结果校准 `quality_score` 权重，避免把启发式分数当真理。
- 不让 pending management 或 entry review 自动下单/自动撤单，继续保持“AI 建议、人工确认、系统记录”的边界。

## Prompt / 观测层：暂缓

prompt 长度已经独立记录到 `ai_prompt_metrics`。后续可以根据统计表再决定是否优化：

- 记录 Gemini 实际 token usage（如果 API response 提供）。
- 增加 prompt 膨胀告警。
- 统计 monthly_goal、pending_orders、trade_review、market_data 等分段长度。

这些属于观测优化，优先级低于先跑交易策略数据。
